Phase 2 — V1 in TWO passes: 2A Structure Frame → 2B Data Hydration
2A (submit_v1_frame): structural skeleton + NARRATIVE SPINE — H2/H3 headings + empty paragraph slots, table HEADERS (no data), FAQ QUESTIONS + empty answer slots, link anchors mapped to the research brief, image slots, CTA placement, checklist, thesis, and a narrative_spine (core_thesis, callback_setup, section_transitions keyed by heading text, 3-5 voice_reminders). No prose, no numbers. JS validates the frame, then 2B — SEQUENTIAL HYDRATION ("the Relay Race", section-level — ~20-25 calls): one hydrate_content_block call for the Direct Answer, ONE per H2 section (batching ALL that section's + its H3s' paragraph ids), one isolated call per table, one per FAQ answer, one for the CTA hook — in article order, each carrying the spine + a rolling draft of everything written so far; the tool returns a `nodes` array (one object per requested id) and JS stitches each into a clone of the locked frame by targetId. Table rows = { cells, sourceReferenceId } citing a Phase 1 factClaim; unsupported rows dropped, never invented. The frozen frame is never mutated; structure physically cannot drift. Phase 3 failures are repaired by surgical micro-calls (submit_v1_corrections) that touch only the failing nodes — the V1 is never regenerated. The cached master-context block (4.3A/4.3B/4.4 + research + thesis) is replayed on every relay call.

Phase 5 — Voice Draft (Option E, section-level relay race)
Claude re-voices the frozen V1's PROSE nodes (Direct Answer, body paragraphs, FAQ answers, CTA hook) as a SECTION-LEVEL RELAY RACE — same pattern as Phase 2B: one revoice_content_block call for the Direct Answer (which also returns the voice_architecture: core conviction, friction setup, callback phrase, pacing strategy, thesis reinforcement), ONE call per H2 section (batching ALL that section's + its H3s' paragraph ids), one call for ALL FAQ answers, one for the CTA hook — in article order, each carrying the FULL rolling V2 draft + the current V1 prose for those nodes. (No per-table call — table data is frozen and disallowed for re-voicing per §5.2/5.3.) The cached master-context block is the SAME one Phase 2 uses (Phase 5 reads it); the voice rules / AI-Tell kill list / intent layer / optional gold-standard exemplars / the "your tone anchor is always 4.3A/4.3B — never the rolling draft" rule ride in a second uncached system block. Each call returns a `nodes` array (one object per requested id, each with data_anchors_to_preserve + a pacing_directive [short punch / standard expansion / list extraction] + transition_logic + the rewrite). JS validates every targetId against the allowed prose-node ids (5.2/5.3), drops orphans/disallowed (F13), and merges the kept rewrites onto a deep clone of V1 → V2. Per-block F2/F3/F6/F13/F14 retry handling (un-re-voiced nodes keep their V1 prose); zero usable rewrites across the whole relay → RED escalate before Phase 6. The frozen V1 bones are never touched. Forensic critique (5.5) + the 6 voice quality checks (5.6) = Milestone 4.
