Phase 2 — V1 in TWO passes: 2A Structure Frame → 2B Data Hydration
2A (submit_v1_frame): structural skeleton + NARRATIVE SPINE — H2/H3 headings + empty paragraph slots, table HEADERS (no data), FAQ QUESTIONS + empty answer slots, link anchors mapped to the research brief, image slots, CTA placement, checklist, thesis, and a narrative_spine (core_thesis, callback_setup, section_transitions keyed by heading text, 3-5 voice_reminders). No prose, no numbers. JS validates the frame, then 2B — SEQUENTIAL HYDRATION ("the Relay Race"): one focused hydrate_node call per prose/table node, in article order, each carrying the spine + a rolling draft of everything written so far; JS stitches each node into a clone of the locked frame by targetId. Table rows = { cells, sourceReferenceId } citing a Phase 1 factClaim; unsupported rows dropped, never invented. The frozen frame is never mutated; structure physically cannot drift. Phase 3 failures are repaired by surgical micro-calls (submit_v1_corrections) that touch only the failing nodes — the V1 is never regenerated. The cached master-context block (4.3A/4.3B/4.4 + research + thesis) is replayed on every relay call.
