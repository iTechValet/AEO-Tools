## ARCHITECTURAL MODEL — AEO TOOL

This tool is built on the same proven patterns as the AEO Article Creation Tool. Do not reinvent what already works.

**What carries over directly from the AEO tool:**
- Single HTML file hosted on GitHub Pages
- Cloudflare Worker proxy (`anthropic-proxy.gerek.workers.dev`) for Claude API calls — reuse as-is
- Google Drive fetch pattern for 4.3A and 4.3B by file ID — same key, same pattern
- Step-by-step progress display (green = done, spinner = running, red = real error only)
- JS owns all calculations — Claude writes prose only
- VA-facing UI — no exposed keys, no terminal, no code

**What is new (the actual build work):**
- 13 API calls to 4 external platforms (ChatGPT, Gemini, Perplexity, Grok)
- Claude Haiku parser with heuristic fallback
- Google Sheets writer via service account
- AI Visibility Score engine (JS only)
- Puppeteer PDF builder
- Google Drive write for PDF delivery
- GitHub Actions trigger mechanism (Phase 1: manual via Cloudflare dispatch proxy; Phase 3: cron)

## ARCHITECTURE OVERVIEW

```
CLIENT CONFIG (JSON, one per client)
        ↓
VA TRIGGERS RUN (index.html → Cloudflare Worker dispatch → GitHub Action)
        ↓
RUNNER (Node.js in GitHub Actions — orchestrates everything)
        ↓
GOOGLE DRIVE FETCH (4.3A + 4.3B — same pattern as AEO tool)
        ↓
SHEET READER (reads full AI_Visibility_Raw history for trend calculation)
        ↓
13 API CALLS — serial, one at a time (ChatGPT, Gemini, Perplexity/sonar, Grok)
        ↓
CLAUDE HAIKU PARSER per run (strict JSON schema + heuristic fallback)
        ↓
SCORE ENGINE (JS calculates AI Visibility Score — never Claude)
        ↓
GOOGLE SHEET WRITER (AI_Visibility_Raw + AI_Visibility_Summary tabs)
        ↓
GITHUB ARCHIVER (raw JSON backup — [YYYY-MM]/[client_id]_raw.json)
        ↓
CLAUDE SONNET NARRATIVE ENGINE (report written in Gerek's voice)
        ↓
PUPPETEER PDF BUILDER (named [client_id]_AI_Visibility_[YYYY-MM].pdf)
        ↓
GOOGLE DRIVE WRITER (PDF → client's "Monthly Reports" folder)
        ↓
[PHASE 3] EMAIL DELIVERY (PDF link, simple template)
        ↓
LOOKER STUDIO DASHBOARD (manual setup — reads Sheet automatically)
```

**Key architectural laws:**
- The AI Visibility Score is calculated by JS. Claude NEVER calculates or interprets the score.
- Claude receives the score, trend, and benchmark as pre-calculated facts and writes narrative around them.
- Dates are never calculated by Claude. Pass `run_date`, `report_period`, and `client_month` as facts.
- API calls are serial. Never parallel. Clean error isolation per run.

## RESPONSE PARSER — CLAUDE HAIKU + HEURISTIC FALLBACK

**Primary:** Claude Haiku (`claude-haiku-4-5-20251001`) extracts structured fields from each raw response.

**Parser prompt architecture (strict JSON enforcement):**
- System prompt specifies rigid JSON schema
- Three few-shot examples: one clear positive, one clear negative, one ambiguous/mixed
- Explicit enum lists for every enum field (e.g., `sentiment_signal: ["positive", "neutral", "negative", "mixed"]`)
- Zero conversational filler permitted in output
- JSON mode enforced

**Fields extracted per run:**
```js
{
  brand_mentioned: boolean,
  brand_recommended: boolean,
  brand_position: "first" | "second" | "third" | "other" | "not_mentioned",
  competitor_mentioned: boolean,
  competitor_recommended: boolean,
  sentiment_signal: "positive" | "neutral" | "negative" | "mixed",
  cited_urls: string[],
  authority_figure_named: boolean,   // Run 3 only
  parser_status: "llm" | "heuristic"
}
```

**Fallback:** If Haiku call fails or returns malformed JSON, heuristic regex attempts the 3 score-critical fields only:
- `brand_mentioned` — business name present in response text
- `brand_recommended` — recommendation language present near business name
- `sentiment_signal` — positive/negative keyword matching

**Threshold rule:** If 4 or more runs in a single month log `parser_status: "heuristic"`, the tool completes the run but adds a ⚠️ flag to the PDF and Sheet row. VA delivers the report and alerts Gerek that parser maintenance is needed. Run does not stop.

## AI VISIBILITY SCORE ENGINE (JS ONLY — NEVER CLAUDE)

Composite score 0–100. Calculated entirely in `score-engine.js`. Claude never touches this math.

```
Mention Rate     (40%) = brand_mentioned runs / brand-trackable runs × 100
Recommendation   (35%) = brand_recommended runs / all runs × 100
Sentiment Score  (25%) = positive sentiment runs / all runs × 100

Final Score = (mention_rate × 0.40) + (recommendation_rate × 0.35) + (sentiment_score × 0.25)
```

**Trend arrow:** Current month score minus prior month score. Pulled from `AI_Visibility_Summary` tab by `sheet-reader.js` before the run starts. JS calculates — no human involvement.

**Benchmark ranges by client_month (displayed alongside score):**
- Months 1–3: Expected 10–25. "Foundation building phase."
- Months 4–6: Target 20–40. "Content compounding."
- Months 7–12: Goal 35–60+. "Recommendation territory."

## RETRY LOGIC (ALL PLATFORMS)

- 5 attempts per run with exponential backoff (not fixed 30-second intervals)
- Backoff schedule: 30s → 60s → 120s → 240s → 480s
- On failure after 5 attempts: log `platform_available: false`, write "data unavailable" to Sheet, continue
- Never crash. Never require human intervention. Never a single point of failure.
- Grok is the most likely to be unavailable — graceful fallback is mandatory for Runs 1D and 5B

## CLAUDE NARRATIVE ENGINE (CLAUDE SONNET — PROSE ONLY)

Claude receives exactly this context. Nothing more, nothing less.

**Inputs passed to Claude:**
1. All raw responses from current month's 13 runs (raw_response fields — labeled by run_id)
2. Full AI_Visibility_Summary history — all prior months (no arbitrary cutoff)
3. AI Visibility Score for this month (pre-calculated by JS)
4. Score last month (read from Sheet by sheet-reader.js)
5. Trend direction — UP / DOWN / FLAT (pre-calculated by JS)
6. Benchmark range for this client_month (pre-calculated by JS)
7. client_month integer
8. run_date and report_period (never let Claude calculate dates)
9. 4.3A content (fetched from Google Drive at runtime)
10. 4.3B content (fetched from Google Drive at runtime)

**Claude's only job:** Write the narrative. All numbers, scores, dates, trends, and benchmarks arrive as pre-calculated facts.

**Report sections (in order):**
1. Executive Summary — 2–3 sentences. The verdict.
2. What We Did This Month — content, milestones, actions. Concrete. Numbers.
3. What the AI Platforms Said — one paragraph per platform. Framed against last month.
4. Where We Stand — score, trend, benchmark context.
5. What's Coming Next — one specific next action. Justifies the retainer.
6. On Our Radar — one forward signal to watch.
7. Closing — Gerek Allen, Founder — iTech Valet signature block.

**Run 6 (competitor intelligence) is NEVER passed to Claude's narrative prompt. Sheet only.**

**Voice rules (from 4.3A and 5.1 Master Prompt — non-negotiable):**
- 5th grade reading level. Casual, direct, warm. Never corporate.
- Short paragraphs. One idea per paragraph. No sentence over 20 words.
- Active voice only.
- Forbidden terms: deployed, clusters, intent layers, pillars, semantic, entity, trunk, branch, schema, backlink, keyword, SEO, Tier 1/2/3.
- No competitor names by name — "the top name in your market" only.
- Drift Detection: Replace {business_name} with a competitor. If it still reads accurate, rewrite.

## VA-FACING UI (index.html)

Single HTML file. Hosted on GitHub Pages. Same pattern as AEO tool.

**UI flow:**
1. VA opens `index.html`
2. Selects client from dropdown (populated from `/clients/*.json` files)
3. Hits "Run Report"
4. UI sends POST to Cloudflare Worker `/trigger-report` endpoint with `client_id` + internal access key
5. Worker validates access key, fires `repository_dispatch` to GitHub with `client_id` as payload
6. GitHub Action picks up the event and runs `runner.js --client=[client_id]`
7. VA's browser shows "Run triggered — check back in ~10 minutes" confirmation
8. VA can close the tab immediately — execution is on GitHub's servers, not the browser
9. On completion, PDF is in Google Drive. VA downloads and sends to client.

**UI/UX rules (inherited from AEO tool):**
- Green = good/expected
- Red = real problem only — never use red for expected or normal states
- Progress display shows each step as it completes

**VA never enters API keys. Ever.**

## LOCKED DECISIONS

These are final. Do not re-litigate in build sessions.

1. **One tool, all clients.** Config file is the only thing that changes per client.
2. **JS calculates the score. Claude writes the story.** Never cross this line.
3. **13 runs, 4 platforms. Serial execution.** Never parallel. Perplexity uses sonar model specifically.
4. **Exponential backoff. 5 retries. Never crash.** Unavailable = log and continue.
5. **Four tabs per client sheet. All tool-written. Zero manual input.** Tab names identical across all clients. Both tools target by name, never by position.
6. **GitHub is the backup.** Raw JSON every run. PDFs go to Google Drive.
7. **Run 6 is internal only.** Never in client deliverables. Never passed to Claude narrative.
8. **Claude receives full Sheet history every month.** No arbitrary cutoff.
9. **Looker Studio is desktop only.** Note in delivery email. Not built — manual setup after Phase 1.
10. **Dashboard shows data. Report tells the story.** Two jobs. Never overlap.
11. **Dates are never calculated by Claude.** Pass run_date, report_period, client_month as facts.
12. **Execution runs in GitHub Actions — not the browser.** Browser is trigger only. VA can close tab immediately.
13. **Claude Haiku for parsing. Claude Sonnet for narrative.** Two separate calls. Two separate jobs.
14. **Parser fallback threshold: 4+ heuristic runs = ⚠️ flag + continue.** Never stop the run.
15. **Master Sheet template governs onboarding.** New client = duplicate template. Never blank sheet.
16. **Cowork is not part of this build.** Phase 3 trigger is GitHub Actions cron. Cowork may be evaluated after 20+ successful runs.
