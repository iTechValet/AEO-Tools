V4.1.6 IMAGE PANEL — REMOVE SLIDE-LEFT, ADD INTERNAL SCROLL, CHEVRON-COLLAPSE ON VALIDATE — May 15, 2026
────────────────────────────────────────────────
Four surgical fixes to the left-column image panel. The V4.1 sliding-drawer
animation is GONE — the slide felt jarring once the panel started auto-
collapsing on validation. Replaced with a quieter opacity + max-height
transition + an internal scrollbar so the image grid is browsable even on
8-image articles. NO pipeline-logic changes.

1. LEFT COLUMN — STICKY VIEWPORT-HEIGHT SCROLL CONTAINER.
   Selector: .left-col
   Old: overflow-y:auto;
   New: overflow-y:auto; height:100vh; position:sticky; top:0;
   Effect: the column now claims exactly the viewport height and scrolls
   its own contents (phase cards + image panel) inside that frame. The
   page no longer extends downward when the operator paastes 8 image URLs.
   .right-col is untouched (it still flows normally inside the grid).

2. IMAGE GRID + PHASE 7 INPUTS — INTERNAL SCROLL, NO SLIDE.
   Selectors: #imgGrid, #p7Inputs
   Old base state:
     transform:translateX(0); opacity:1;
     max-height:1200px;
     overflow:hidden;
     transition: transform .28s, opacity .22s, max-height .32s;
     will-change: transform, opacity, max-height;
   New base state:
     opacity:1;
     max-height:520px;
     overflow-y:auto;
     transition: opacity .22s, max-height .32s;
   Old collapsed state (.collapsed / .images-collapsed):
     transform:translateX(-110%); opacity:0; max-height:0;
   New collapsed state:
     opacity:0; max-height:0; overflow:hidden;
   Why: the slide-left animation was meant to read as "drawer closing,"
   but with the auto-collapse on validate it kept catching the operator's
   peripheral vision on every successful run. The new collapse is a quiet
   fade + height drop. The 520px max-height + overflow-y:auto on the open
   state gives the panel an INTERNAL scrollbar (fits about two image
   cards; operator scrolls inside the panel for the rest) instead of
   trying to fit all 8 cards into the layout at once.

3. validateImages() — DIRECT CLASS APPLICATION, NOT togglePhase7ImgPanel.
   Removed:
     togglePhase7ImgPanel({ collapse: true });
   Added:
     const _panel = document.getElementById('imagePanel');
     if(_panel){ _panel.classList.add('images-collapsed'); }
     const _chevron = document.getElementById('imgPanelToggle');
     if(_chevron){ _chevron.setAttribute('aria-label','Expand image panel'); }
   Why: routing through togglePhase7ImgPanel meant the GREEN-gate collapse
   went through the same code path as the operator's chevron click, which
   was overkill (and confused the class-toggle logic in step 4). Directly
   adding .images-collapsed keeps the auto-collapse path self-contained.
   The aria-label flip keeps the screen-reader copy in sync with the
   visual state.

4. togglePhase7ImgPanel() — STOP DOUBLE-TOGGLING .images-collapsed.
   Old:
     panel.classList.toggle('collapsed', wantCollapsed);
     panel.classList.toggle('images-collapsed', wantCollapsed);
   New:
     panel.classList.toggle('collapsed', wantCollapsed);
     if(!wantCollapsed) panel.classList.remove('images-collapsed');
   Why: with the V4.1.4 logic, clicking the chevron after an auto-collapse
   added BOTH classes on the way back open (because they were toggled in
   lockstep), which left the panel collapsed because .images-collapsed was
   re-added the moment .collapsed was removed. New logic: the manual
   chevron path toggles ONLY .collapsed; expanding the panel ALSO strips
   .images-collapsed so an auto-collapsed panel re-opens cleanly. Both
   classes still share the same CSS collapse-state selectors so either
   one alone is enough to collapse.

  • validClusterUrls firewall — untouched.
  • Universal Sanitizer — untouched.
  • Ghost CTA default — untouched.
  • AEO_Run_Data telemetry — untouched.

V4.1.5 FINALIZE AUTO-DOWNLOADS + APPS SCRIPT TELEMETRY WEBHOOK — May 15, 2026
────────────────────────────────────────────────
Two operator-experience fixes on top of V4.1.4. NO pipeline-logic changes —
Phase 7 still runs identically, telemetry still fires at the same call site,
just with a saner delivery contract.

1. FINALIZE → AUTO-DOWNLOAD BOTH FILES.
   Selector: function finalizePhase7()
   After the Checklist + Final TXT download buttons are enabled, the function
   now calls dlChecklistHTML() + dlFinalTxt() directly. The operator no longer
   has to click the two ↓ buttons after every successful Finalize — both files
   land in Downloads/ automatically.
   Manual fallback: the buttons stay enabled below the iframe, so if the
   browser throttles the second a.click() (some Chromium builds rate-limit
   consecutive programmatic downloads), the operator can still click either
   button.
   Status line: replaced
     "✓ Finalized — N chars. Downloads ready below. (Any further formatting
      change will require re-finalizing.)"
   with
     "✓ Finalized — both files downloaded automatically. Use the buttons
      below if either download failed."

2. TELEMETRY EXPORT → APPS SCRIPT WEB APP.
   Selector: async function exportTelemetryToSheets()
   Old path: POST directly to the Sheets v4 REST endpoint
     https://sheets.googleapis.com/v4/spreadsheets/<id>/values/<range>:append
     with valueInputOption=USER_ENTERED + insertDataOption=INSERT_ROWS,
     authed by ?key=<Drive API key>.
   Failure modes (why we changed): the API-key auth path only works when the
   sheet is set to "Anyone with the link can EDIT" — most client 4.2 sheets
   are restricted, so the call 401s. And the application/json POST triggers
   a CORS preflight the public Sheets endpoint will answer, but no longer
   reliably with the new V4.1 cost-stat call pattern.
   New path: POST to a deployed Apps Script Web App
     https://script.google.com/macros/s/AKfycbwM3wEawyByZhfD3vTRoI7xXIoPr5LnRLHKDQ1FGTsgBfOrmsc9F9SpuMpTnUyR9eTU/exec
   The web app's doPost(e) opens the client's 4.2 sheet by sheetId (passed in
   the payload) and appends the row to AEO_Run_Data server-side, running with
   the deployed identity (the script owner has edit access to every client
   sheet). The browser side no longer needs a Drive API key for telemetry —
   the keyDrive field still exists for other Drive operations, but it is no
   longer read by this function.
   CRITICAL: Content-Type: text/plain (NOT application/json).
   Apps Script web apps DO answer cross-origin POSTs with no preflight, but
   ONLY when the Content-Type is text/plain (or one of the other "simple"
   CORS-safe values). An application/json POST triggers a CORS preflight
   OPTIONS request that Apps Script does not answer with the right headers,
   so the fetch fails before doPost() ever runs. The body is still valid JSON
   — only the header reads text/plain. doPost() does
     JSON.parse(e.postData.contents)
   to recover the object.
   Payload shape:
     {
       sheetId,                       // client.sheet42
       date,                          // "YYYY-MM-DD" from sessionTelemetry.startTime
       client,                        // S.client.name or "—"
       topic,                         // S.topic or "—"
       runTime,                       // fmtElapsedMMSS(elapsedMs)
       cost,                          // "$N.NNN" from calculateTotalCost()
       tokens,                        // String(totalTelemetryTokens())
       status                         // sessionTelemetry.status or "Success"
     }
   Response shape (web app contract):
     { result: 'success' }                               → log ok
     { result: 'error', message: '<server-side reason>' } → log warn with the message
   Log lines:
     missing sheet ID → "Telemetry export skipped — no sheet ID for this
                         client." at warn level (replaces the old info-level
                         "Tab 'AEO_Run_Data' not found." copy, which was
                         misleading on the API-key path).
     success         → "Telemetry exported → AEO_Run_Data (date · client ·
                        topic · cost)." at ok level.
     server error    → "Telemetry export failed: <message>" at warn level.
     thrown          → "Telemetry export error: <message>" at warn level.
   Never blocks the finalize flow — every error path either returns silently
   or logs and exits without throwing.

PIPELINE GUARANTEES UNCHANGED.
  • Phase 1–7 prompt + tool wiring — untouched.
  • validClusterUrls firewall — untouched.
  • Universal Sanitizer — untouched.
  • Ghost CTA default — untouched.
  • sessionTelemetry capture points (Gemini + Claude token deltas at every
    cost-stat update site) — untouched. Only the export transport changed.

(May 12–13, 2026. Two-day window. The 7-phase pipeline as shipped in M5 went
through its first end-to-end live runs against real iTV topics, and the
architectural failure modes that surfaced — image-gate flapping, ToC duplicate
FAQ entries, image slots injecting into FAQ headings, conclusion content
landing in content_sections, "Gerek Allen, Founder, Founder" byline duplication,
consecutive duplicate paragraphs, Phase 3B false-positive thrashing on
conversational language, full-pipeline aborts when a single section
hallucinated — were patched in-session via a schema refactor, a Phase 3B
recalibration, the Save State rollback architecture, and assembly-time
delivery polish. Twelve commits across the two days, all live on main, all
syntax-checked, with multiple end-to-end live runs verifying each before the
next layer of changes shipped. Build target unchanged: AEO-Article-Creation/
aeo_v4.html. The 7-phase pipeline shape is unchanged; the schema, the Phase
3B contract, and Phase 2B's retry topology all upgraded.)

- SCHEMA REFACTOR — Phase 2A frame, V1_FRAME_TOOL.input_schema, Phase 3A
  validation, Phase 7A assembly (commit 04e8c55, May 13). The frame's
  generic `sections` array + `faq` array are gone. The frame JSON now
  ships three dedicated top-level keys:
    • content_sections   — standard H2 sections only. Every entry gets a
                           ToC entry + an injected image. NO FAQ, NO
                           conclusion in this array. Min 4.
    • conclusion_section — { id, h2, paragraphs[] }. ALWAYS required.
                           Own top-level key with its own id. NO image
                           slot, NO image rendering, at any phase. Ever.
    • faq_section        — { id, h2, intro_paragraphs[], qa_pairs[{id,
                           question, answer{}}] }. ALWAYS required. Own
                           top-level key. NO image slot, NO image
                           rendering, at any phase. Ever.
  V1_FRAME_TOOL.input_schema requires all three; the Phase 2A prompt
  spells out the shape and the "ALWAYS required, both intro_paragraphs
  AND qa_pairs" rule for faq_section. Phase 3A adds hard validation:
  conclusion_section_missing, faq_section_missing, conclusion_paragraphs_
  empty, faq_intro_empty, faq_answer_empty, faq_qa_count (≥5),
  conclusion_section_h2_empty, faq_section_h2_empty.
  Phase 7A assembly renders DETERMINISTICALLY from these three keys with
  ZERO text-matching anywhere: DA → Last Updated + byline → ToC (sourced
  strictly from content_sections H2s + conclusion_section.h2 +
  faq_section.h2) → each content_section (H2, image, paragraphs, h3s,
  tables) → conclusion_section (H2 + paragraphs only — no image, no
  table, ever) → faq_section (H2, intro_paragraphs as <p>, then each
  qa_pair as H3 + answer prose — no image, ever) → CTA. Every previous
  toLowerCase().includes() / regex used to identify FAQ or conclusion
  inside the old generic `sections` array is gone.
  Downstream: Phase 2B hydration relay queues a dedicated conclusion
  block + a FAQ block (intro paragraphs + every qa_pair answer batched);
  Phase 5 voice relay + Phase 6 critique relay both iterate through
  conclusion paragraphs + FAQ intro paragraphs alongside content-section
  paragraphs and qa_pair answers; the conclusion job in Phase 6 is the
  lastSectionJobIdx so QC2 (callback verifier) + QC6 (semantic anchor)
  literally fire on the conclusion section.
  Resolves three live-run failure modes in one structural change:
  missing conclusions slipping past validation, duplicate FAQ ToC entries,
  and images injecting after FAQ headings.

- SECTIONAL FACT ALLOCATION — Phase 2A frame field, Phase 2B per-section
  factClaim scoping (commit 09250e1, May 13). The Phase 2A frame now
  produces, on EVERY section (content / conclusion / FAQ), a required
  assigned_facts array of 1-3 factClaim IDs drawn from the Phase 1
  research brief. Phase 2B hydration calls receive ONLY the 1-3
  factClaims assigned to the section being written — not the full
  factClaim menu. JS validates that each assigned id is real (present in
  Phase 1 factClaims) AND, for quantitative-table rows, that the row's
  sourceReferenceId is in the parent section's assigned_facts pool.
  Eliminates cross-section factClaim confusion (e.g. a stat about "agency
  conversion rates" being cited inside a section about "site speed") and
  reduces per-call token pressure (the section sees ~3 factClaims, not
  the full ~20-claim brief).

- INLINE CITATION TOKENS — Phase 2B prompt + Phase 7A render-time strip
  (commit 09250e1, May 13). Phase 2B's prompt requires Claude to append
  the citation token [factClaim_NN] in brackets IMMEDIATELY after any
  specific number, percentage, dollar amount, timeframe, or named study
  used in prose or table cells. Example: "Agencies waste significant
  budget [factClaim_02] on tactics that don't build entity trust." The
  token MUST reference one of THIS section's assigned_facts. Forces
  Claude to verify the claim before completing the sentence — and JS
  inline-validation rejects the section if a token references a
  factClaim NOT assigned to it (Save State retry kicks in). Phase 7A
  mdInline strips every [factClaim_NN] token before parsing, so tokens
  never appear in the rendered preview iframe, the final .txt, the
  operator HTML source view, or the BlogPosting / FAQPage / Breadcrumb
  schema JSON. Tokens exist only between Phase 2B writing and Phase 7A
  rendering — purely internal anchors.

- FRAMEWORK LABELS — Phase 2A frame field + Phase 2B verbatim injection
  (commit 09250e1, May 13). The Phase 2A frame outputs a top-level
  framework_labels JSON object — a flat string→string map of every
  custom multi-part structural label Claude invents in the article
  (numbered stages, named phases, pillars, layers, steps). Example:
  {"Stage 1":"Dormancy","Stage 2":"Displacement","Stage 3":"Invisibility"}.
  Empty {} if the article has no custom framework. The map is LOCKED
  once written. Phase 2B injects it verbatim into every hydration call
  as a "use these exact strings — do not alter, abbreviate, or
  paraphrase" reference. Prevents internal-consistency failures (e.g.
  "Stage Two dormancy" appearing when Stage One was named "Dormancy"
  in section 1). Phase 3A validates the field's presence and that
  every value is a string.

- TYPED TABLES — Phase 2A frame field + Phase 2B per-type constraints
  (commit 09250e1, May 13). The Phase 2A frame now declares table_type
  on every table, enum {quantitative, conceptual}. QUANTITATIVE tables
  enforce the CoT fact-anchor justification schema: Claude must quote
  the supporting factClaim's exact phrase in fact_anchor_justification
  BEFORE writing the cells, and sourceReferenceId must come from the
  parent section's assigned_facts. Rows without supporting facts are
  sentinel-tagged "DELETE_ROW_UNSUPPORTED" and JS guillotines them
  post-stitch. CONCEPTUAL tables organize narrative comparisons /
  frameworks / structural info with no factClaim binding required —
  cells only. The Phase 3A minimum table count (≥3 across all content
  sections) is maintained; conceptual tables count toward the
  minimum. HYDRATE_CONTENT_BLOCK_TOOL.input_schema was relaxed so
  sourceReferenceId + fact_anchor_justification are per-table-type
  optional (JS enforces them only for quantitative).

- PHASE 2B NUMERICAL ABSOLUTE RULE — Phase 2B uncached instr block
  (commit 09250e1, May 13). The hydration prompt's NUMERICAL ABSOLUTE
  RULE: Claude is strictly forbidden from inventing specific numbers,
  percentages, timeframes, or dollar amounts unless they appear
  verbatim in one of the section's assigned factClaims. Explicit
  escape hatch: describe concepts qualitatively when no assigned
  factClaim supports a specific figure (e.g. "requires significant
  capital expenditure" instead of "$30,000", "over several months"
  instead of "3-6 months"). Combined with the citation-token rule
  this gives a closed loop: every specific figure must be anchored
  to an assigned factClaim, or it must not be there.

- SAVE STATE ROLLBACK — Phase 2B per-section snapshotting + Phase 3B
  micro-retry (commit 09250e1 + 45f1d0c, May 13). Phase 2B now
  snapshots the article JSON + rolling draft state AFTER each
  successfully hydrated section passes inline validation (all
  required ids non-empty AND no cross-section citation token). If a
  section fails inline validation OR fails Phase 3B's hallucination
  sweep, JS rolls the article back to the last good snapshot and
  retries ONLY that section — up to 2 Save State micro-retries with
  a targeted warning (the previous failure reason / the offending
  text Gemini flagged) fed back into the per-section prompt. Cost
  of a section retry is roughly the cost of one Phase 2B section
  call (~$0.10-0.15) — NOT the cost of regenerating the entire
  article (~$1.50+). If a section still fails after 2 retries on
  EITHER stage, the pipeline emits an immediate kill-only RED
  escalation via buildKillOnlyRed (Stage / Section H2 / failure
  reason / "This run has been stopped after 2 retry attempts. Start
  a new run or move to the next topic.") and halts — no resume, no
  fix UI. The .esc-msg CSS got white-space:pre-wrap so the
  multi-line RED renders structured in the escalation panel.

- PHASE 3B RECALIBRATION — buildPhase3BPrompt rewritten as a data-
  integrity-only sweep with FLAG/IGNORE lists (commit 09250e1 +
  45f1d0c, May 13). Phase 3B's prompt is stripped to one job: did
  Claude fabricate a specific statistic / percentage / dollar amount /
  named study and present it as a sourced fact — when it does NOT
  appear in the Phase 1 factClaims or the 4.3B client identity
  document? Explicit FLAG list:
    • Specific percentages or fractions presented as research-backed
      facts not present in factClaims
    • Specific dollar amounts presented as facts not present in
      factClaims
    • Named research studies, reports, or case studies not in
      factClaims
    • Named companies, software tools, or agencies not in 4.3B or
      factClaims
  Explicit IGNORE list (the prompt says "Do not flag, do not mention"):
    • Conversational timeframes ("in six weeks," "within a quarter,"
      "a few months")
    • Written-out numbers used rhetorically ("eighteen clients," "a
      dozen practices")
    • Qualitative comparisons ("most businesses," "many practices,"
      "the majority")
    • General duration or frequency language
    • Any prose that sounds specific but makes no claim to be sourced
      data
  Phase 3B is no longer asked to check anchor text, structural
  elements, framework-label consistency, internal-link placement, or
  writing quality — those belong elsewhere. The response shape is
  typed: { verdict:"PASS"|"FAIL", hallucinations:[{nodeId,
  offendingText, location:"table_row"|"prose"}] }. Routing:
  table_row hallucinations → JS guillotine (the matching row is
  deleted from the article JSON, pipeline continues, no Claude
  call); prose hallucinations → Save State micro-retry of the
  containing section (the section's prose is cleared, a new
  hydrate_content_block call fires with Gemini's flagged text as
  the retry warning, then 3B re-runs). Cap: 2 micro-retries per
  section. The exhaustive-audit semantic checker + the surgical-fix
  loop are GONE from Phase 3B — surgical fix remains only for Phase
  3A structural failures (2-round cap, unchanged).

- BYLINE TITLE EXTRACTION FIX — extractBrandingFrom43B + byline
  assembly guard (commit e0efe2a, May 13). Symptom from live runs:
  bylines rendering as "Gerek Allen, Founder, Founder" or "Gerek
  Allen Author Title: Founder." Root causes were:
    (1) The authorName main regex captured greedily across the comma
        in a same-line "Author: Name, Title" pattern, sweeping the
        title into the captured name.
    (2) On a same-line "Author: Name Author Title: Title" pattern,
        the authorName regex captured the entire trailing string,
        including the label "Author Title:".
    (3) The captured authorTitle could leak surrounding markdown bold
        (e.g. **Founder**) when 4.3B used `**Author Title:**` styling.
  Fix:
    • authorName main regex now stops at: a COMMA, the literal
      "Author Title:" on the same line, pipe, newline, or end. The
      Byline / Written-by fallback intentionally keeps no comma stop
      (it stays a verbatim override).
    • Both name and title regexes consume any leading / trailing **
      around the value or the label.
    • The primary authorTitle regex captures [^\r\n|*]+? — asterisks
      excluded — so markdown bold around the value cannot leak in.
    • Belt-and-suspenders scrub strips any residual "Author Title:" /
      "Title:" prefix and any surrounding asterisks from the captured
      title.
    • Byline assembly in assemblePhase7HTML adds a duplicate-title
      guard: if authorTitle already appears at the end of authorName
      (case-insensitive, with or without a comma separator), the
      title is NOT appended again. Eliminates the doubled-title
      output in all observed forms.

- DUPLICATE PARAGRAPH DETECTION — assemblePhase7HTML render guard
  (commit e0efe2a, May 13). Symptom from live runs: "That playbook
  is dead." appearing twice back-to-back in one section. assemble-
  Phase7HTML now runs a CONSECUTIVE-duplicate guard during rendering:
  for each paragraph in content_sections, content_sections.h3sections,
  conclusion_section, and faq_section.intro_paragraphs, the trimmed
  prose is compared to the IMMEDIATELY preceding paragraph's trimmed
  prose; on exact match the second is skipped (not rendered). The
  tracker resets at every H2 content section, every H3 inside a
  content section, the conclusion_section, and the faq_section.intro_
  paragraphs — so non-consecutive repeats elsewhere in the article
  still render. qa_pair answers are NOT deduped (per spec — content
  sections, conclusion, FAQ intro only). The skip count is logged
  yellow at the end of assembly when > 0. The article JSON is NOT
  mutated; this is render-time only.

- CONCLUSION-SECTION INTEGRITY CHECK — Phase 2B end-of-relay
  diagnostic (commit e0efe2a, May 13). Symptom from live runs: the
  Phase 2A frame occasionally puts conclusion-style content into a
  content_sections entry (with a heading like "The Bottom Line" or
  "Where This Leaves You") rather than into the dedicated
  conclusion_section, leaving conclusion_section empty and the
  conclusion content rendered as a normal content section (with an
  injected image). After executePhase2B's main loop finishes, JS
  now verifies that conclusion_section is a top-level object AND
  has ≥1 paragraph with non-empty prose. If missing or empty, JS
  scans content_sections for any H2 whose heading text matches a
  conclusion-style label regex AND has hydrated paragraphs. Each
  match is logged as a misplaced node (section index + H2 text +
  hydrated paragraph ids) via warnYellow F14 — diagnostic only.
  JS does NOT auto-move the content; the misplacement is flagged
  so the operator can see which frame attempt produced misplaced
  content, and Phase 3A's conclusion_paragraphs_empty hard fail
  still fires next to force a Phase 2A retry. CURRENT REGEX (as of
  e0efe2a): /\b(conclusion|conclud|bottom\s+line|wrap(?:ping)?\s+
  up|final\s+(?:thought|word|takeaway)s?|in\s+closing|to\s+sum\s+
  up|takeaway|summary|the\s+last\s+word|where\s+this\s+leaves\s+
  you)\b/i — covers ~10 common patterns. NOTE: live-observed
  headings like "The Widening Gap" or "The Verdict" do NOT match
  this regex and will silently NOT be caught by this YELLOW
  diagnostic; expansion is a tracked watch item (see V4.0 KNOWN
  BUGS / WATCH ITEMS below). Phase 3A's structural fail still
  forces the retry whether or not this diagnostic fires.

- IMAGE QUEUE TUNING — fail-fast + back-of-queue final retry
  (image-gen module, current code). ATTEMPT_TIMEOUTS = finalPass ?
  [90000] : [35000, 45000]. Attempt 1 times out at 35 s (fail-fast,
  was 90 s nominal). Attempt 2 (immediate retry) at 45 s. Attempt 3
  pushes the image to the BACK of the queue (runs after every other
  image finishes, at the 90 s timeout). 503 capacity errors and 429
  rate-limit errors on a non-final pass throw {requeue:true} → the
  image is pushed to the back-of-queue for attempt 3 (no separate
  fixed-duration backoff before the requeue; back-of-queue latency
  comes from waiting for all other images to finish). A jittered
  2.5–5 s delay precedes every sequential image API call. Manual
  "↻ regenerate this image" uses finalPass=true (the 90 s single
  attempt, no requeue). Designed to recover from transient Gemini
  capacity blips without compounding them. Total attempts: up to 3
  per image.

      const jitter = Math.floor(Math.random() * 2500) + 2500;

  the dead attempt's queue and clears its URLs (zombie-queue protection).

V4.0 ARCHITECTURAL LOCKS (so far — will be consolidated when V4.0 ships):
  V4-1. JS owns structure; Claude owns prose; Gemini owns research + semantic
        validation. No tool's output is another tool's spec without
        deterministic verification first.
  V4-2. JS assigns ALL IDs post-parse (Claude never writes id fields). LLMs
        are bad at counting — downstream patches target IDs, not array indexes.
  V4-3. Claude returns JSON via Tool Use schemas, never freeform. Every
        response handler runs stripMarkdownWrappers() before JSON.parse()
        then a tolerant parser (Critical Directive #1 — non-negotiable).
  V4-4. Phase 4 Object.freeze is the structural lock — there is no Frame Lock
        / Snapshot Diff / structural-coordinate machinery in V4.0 and it must
        not come back. If the bones can drift, the freeze is broken — fix the
        freeze, don't add a backstop.
  V4-5. Single cached system block per Claude phase, cache_control on the last
        block. Cache hit ratio is a tracked health metric (5-min TTL — slow VA
        review = uncached call = cost spike).
  V4-6. No human reads article content mid-pipeline. Calibration mode = operator
        reviews the finished draft before publish; Production mode = auto-publish
        on all-checks-pass. Promotion is a manual operator decision, never
        auto-triggered by article count.
  V4-7. Every failure has a defined F-code response (F1–F21 in the build plan).

to hunt for the DA boundaries. Roughly 1 in 10 runs Claude varied the output structure

locked. Patched 5 times across V2.5 → V2.8 → V3.0; each patch reactive. JSON eliminates
the root cause: Claude must produce structured data, JS calls JSON.parse(), failure is

VCI — VOICE COMPLIANCE INDEX (JS-BASED, V2.5)
----------------------------------------------
Step 13 counts violations using JS regex. No Gemini call for counting.
Deterministic, instant, zero cost, parse errors architecturally impossible.

jsVCICount(v2Body, forbiddenWordsList) returns 5 raw violation counts:

1. expanded_contractions — word-boundary regex on uncontracted forms
2. long_sentences — split on terminators, count words, flag >15
3. hedging — combined regex of single words + multi-word phrases
4. passive_voice — V2.5 expanded to catch 3-word constructions
5. forbidden_words — universal banned list + 4.3B Terminology Lock terms

JS calculates: rate = totalViolations / (wordCount / 500)
              score = max(0, round(100 - rate))
Verdicts: PASS (90+), FIX (75-89), REWRITE (<75)

OPTIONAL CONVICTION SCORE (small Gemini call):
0-10 subjective tone rating. Non-blocking. Score shows "—" if call fails.

  context using JavaScript string-matching and regex. Every patch (Frame Lock,
  Snapshot Diff, structural coordinates, fuzzy matching, hardRestoreDAZone)

  article in context.

  to give Claude actual context (the same context the chat version had).
  Use strict JSON throughout via Tool Use. Let JavaScript handle deterministic
  checks. Let Gemini do ONE job: semantic fact-check against research brief.
