MILESTONE 2.5 — PHASE 2 REBUILD: TWO-PASS (2A FRAME → 2B HYDRATION) — ✅ BUILT, ⏳ LIVE TEST PENDING
(Architectural correction, May 11, 2026. The single-pass Phase 2 made Claude
do structure + prose + fact retrieval at once → hallucinated numbers in
tables; Gemini caught them in 3B and the old retry loop made it worse by
regenerating the whole article. V3.1 had the right pattern — Step 3 structure,
Step 4 JS validate, Step 5 fill prose. M2.5 restores it and adds a surgical
third pass. Full spec is in AEO_V4_Build_Plan.txt "PHASE 2" section.)

- PRE-STEP (JS): factClaim_01, factClaim_02, … stamped onto S.research.factClaims
  after Phase 1 (Phase 1's Gemini prompt unchanged — post-parse JS step).
- PHASE 2A — STRUCTURE FRAME (Claude → submit_v1_frame): the skeleton ONLY —
  checklist, thesis statement, H2/H3 headings, table HEADERS (no rows), FAQ
  QUESTIONS (no answers), link anchors mapped to research anchorTextTargets,
  image slots, CTA placement. No prose, no numbers. JS assigns section_/h3_/
  table_/link_/img_/faq_/cta_/thesis ids.
- JS FRAME VALIDATION (between 2A and 2B): the "frameOnly" mode of
  validateV1Structure — structural minimums + link discipline + checklist +
  CTA-url + thesis-present, skipping DA/prose/row checks. Fail → re-run 2A
  with feedback, max 2 → else CHK1.
- PHASE 2B — DATA HYDRATION (Claude → submit_v1_hydration): given the validated
  frame + the factClaim menu, returns prose + FAQ answers + table data rows
  keyed by frame node id. Table rows are { cells, sourceReferenceId } — each
  citing a Phase 1 factClaim; a row with no supporting factClaim is DROPPED,
  not invented.
- JS MERGE (mergeHydration): JS — not Claude — splices the hydration into the
  frame by id, creating the da_/para_/ans_ nodes. The merge mutates the frame
  in place → the frame IS the V1. Claude physically cannot move the bones, so
  there's no "did Claude change a heading?" failure mode. Orphan ids dropped
  (F13 YELLOW).
- TABLE-ROW SCHEMA MIGRATED: rows: [["a","b"], …] → rows: [{ cells:["a","b"],
  sourceReferenceId:"factClaim_NN" }, …]. normalizeV1Shape coerces any legacy/
  malformed shape so old data never crashes the pipeline.
- PHASE 3A — NEW CHECKS: every table row carries a sourceReferenceId pointing
  at a real Phase 1 factClaim id; cells align with headers; no empty tables;
  ≥1 non-empty paragraph per section and per H3. A missing sourceReferenceId
  is an immediate structural fail — the Gemini call is skipped that round.
- PHASE 3 RETRY = SURGICAL: on any 3A/3B fail, JS turns each failing node into
  a fix packet ({targetId, problem, context, node}) and makes ONE micro-call
  (Claude → submit_v1_corrections) returning replacements for ONLY those nodes

  regenerated. Max 2 surgical rounds → CHK1. (A non-node-level fail — e.g. a
  count failure, which can only mean the frame regressed — escalates CHK1
  rather than spinning the surgical loop.)
- CACHING: ONE shared cached context block (4.3A+4.3B+4.4+research-with-ids+
  thesis+topic+sheet row) byte-identical across 2A, 2B, and the surgical fix

V4.0 ARCHITECTURAL LOCKS (so far — will be consolidated when V4.0 ships):
  V4-1. JS owns structure; Claude owns prose; Gemini owns research + semantic
        validation. No tool's output is another tool's spec without
        deterministic verification first.
  V4-2. JS assigns ALL IDs post-parse (Claude never writes id fields). LLMs
        are bad at counting — downstream patches target IDs, not array indexes.
  V4-3. Claude returns JSON via Tool Use schemas, never freeform. Every
        response handler runs stripMarkdownWrappers() before JSON.parse()
        then a tolerant parser (Critical Directive #1 — non-negotiable).
  V4-4. Phase 4 Object.freeze is the structural lock — there is no Frame Lock
        / Snapshot Diff / structural-coordinate machinery in V4.0 and it must
        not come back. If the bones can drift, the freeze is broken — fix the
        freeze, don't add a backstop.
  V4-5. Single cached system block per Claude phase, cache_control on the last
        block. Cache hit ratio is a tracked health metric (5-min TTL — slow VA
        review = uncached call = cost spike).
  V4-6. No human reads article content mid-pipeline. Calibration mode = operator
        reviews the finished draft before publish; Production mode = auto-publish
        on all-checks-pass. Promotion is a manual operator decision, never
        auto-triggered by article count.
  V4-7. Every failure has a defined F-code response (F1–F21 in the build plan).
        Pipeline auto-recovers where it can; RED escalations surface to the VA
        as a code + one-line remediation; the operator handles RED. No reactive
        patching.
  V4-8. Client-agnostic from day one — every feature reads client data from
        existing sources (aeo_clients config, 4.3B, 4.2 sheet incl. INTENT_LAYER).
        Must run identically on iTV, Brighton Gold, Bushido, MV with zero
        per-client tool changes. The TOOL ADAPTS TO THE DOC, never the reverse —
        e.g. the CTA-url check reads whatever shape 4.3B uses ("Primary CTA"
        section's "Destination:" line, or a "…CTA URL" field); clients are
        never asked to rename a 4.3B field to match the tool.
  V4-9. PHASE 2 IS TWO PASSES, NEVER ONE. 2A builds the structure frame only
        (no prose, no numbers); JS validates it; 2B fills prose + data rows
        against the frozen frame; JS merges/stitches by id. Every table data
        point must cite a Phase 1 factClaim id (sourceReferenceId) — rows with
        no supporting fact are dropped, not invented. Validation failures are
        repaired by SURGICAL micro-calls that touch only the failing nodes;
        the V1 is never regenerated wholesale. (Collapsing 2A+2B into one call
        is the mistake V4.0's first build made — it made Claude hallucinate
        numbers. Do not re-collapse it.)
  V4-10. PHASE 2B IS A SEQUENTIAL "RELAY RACE" AT SECTION GRANULARITY. One call
        for the Direct Answer, ONE call per H2 section (batching ALL that
        section's + its H3s' paragraph ids), one isolated call per table, one
        per FAQ answer, one for the CTA hook — in article order, each carrying a
        rolling draft of everything written so far + the narrative_spine, via
        hydrate_content_block (returns a `nodes` array). JS stitches each node
        into a clone of the locked frame. Do NOT go back to a single monolithic
        2B call (output-token fatigue, ~50% failure) and do NOT go per-paragraph
        (~63 calls, ~$2). ~20-25 calls is the target.
  V4-11. ANY "WRITE THE WHOLE ARTICLE IN ONE CALL" STEP MUST USE THE RELAY-RACE
        PATTERN. Phase 5 (voice draft) and Phase 6 (forensic critique — the build
        plan's "Phase 5.5") are BOTH section-level relays as of May 12, 2026 (M3.6,
        M4) — a single big Option-E / critique call fatigues on output tokens
        exactly the way the monolithic Phase 2B did. Phase 5: 1× DA → 1× per H2
        section → 1× all FAQ → 1× CTA hook (NO per-table call — table data is
        frozen and disallowed for re-voicing). Phase 6: 1× per H2 section (with a
        sliding window of the neighbours' Phase-5 prose) → 1× all FAQ → 1× DA →
        1× CTA hook. New long-output Claude steps: relay-race them from the start.

REASSEMBLY ORDER (assembleV1FromJSON):
  1. Blog Post Optimization Checklist block (built from checklist object)
  2. --- separator
  3. # H1 (from checklist.h1)
  4. directAnswer prose
  5. <!-- DA_END --> marker (re-inserted by JS for downstream compatibility)
  6. LAST_UPDATED_PLACEHOLDER
  7. TOC_PLACEHOLDER
  8. skeleton (starts at first H2)

PARSING SAFETY:
- Strip markdown code fences if Claude adds them (despite instructions)
- Find first { and last } to handle any preamble/postamble
- Hard fail if JSON.parse() throws — clear error message, retry from Step 3
- Validate three required keys present, directAnswer >= 100 chars, skeleton >= 500 chars
- If validation fails: throw with specific reason, clean retry path

WHY THIS EXISTS:
The pre-V3.0.1 path asked Claude for freeform markdown and used regex (extractLockedDA)
to hunt for the DA boundaries. Roughly 1 in 10 runs Claude varied the output structure
(different checklist heading prefix, missing/extra separator, [H1] suffix variations).
The regex would silently return empty string. Pipeline failed at Step 4 with no DA
locked. Patched 5 times across V2.5 → V2.8 → V3.0; each patch reactive. JSON eliminates
the root cause: Claude must produce structured data, JS calls JSON.parse(), failure is
clean and visible.

VCI — VOICE COMPLIANCE INDEX (JS-BASED, V2.5)
----------------------------------------------
Step 13 counts violations using JS regex. No Gemini call for counting.
Deterministic, instant, zero cost, parse errors architecturally impossible.

jsVCICount(v2Body, forbiddenWordsList) returns 5 raw violation counts:

1. expanded_contractions — word-boundary regex on uncontracted forms
2. long_sentences — split on terminators, count words, flag >15
3. hedging — combined regex of single words + multi-word phrases
4. passive_voice — V2.5 expanded to catch 3-word constructions
5. forbidden_words — universal banned list + 4.3B Terminology Lock terms

JS calculates: rate = totalViolations / (wordCount / 500)
              score = max(0, round(100 - rate))
Verdicts: PASS (90+), FIX (75-89), REWRITE (<75)

OPTIONAL CONVICTION SCORE (small Gemini call):
0-10 subjective tone rating. Non-blocking. Score shows "—" if call fails.

  During final article test on V3.1, multiple architectural failures cascaded:
  Step 8 produced FAILs with invalid coordinate formats and out-of-range
  heading indexes. Step 9 infinite-looped on FAIL_008. Step 10 audit miscounted
  (reported 0 links and 0 FAQ questions when article had 22 links and 6 FAQs).
  Step 11 Frame Lock locked the corrupt audit state. Step 14 Snapshot Diff
  "restored" headings by index against the corrupt snapshot, scrambling H2
  text between sections. Operator killed the run.

  Root cause confirmed: V3.x architecture tries to recreate stateful LLM
  context using JavaScript string-matching and regex. Every patch (Frame Lock,
  Snapshot Diff, structural coordinates, fuzzy matching, hardRestoreDAZone)
  was added to backstop the previous layer's failures. We're rebuilding
  context simulation in JS — and losing.

  Operator confirmed earlier WORKING workflow was manual copy/paste between
  Claude Project chat and Gemini chat. Each chat had full context. Articles
  came out clean. The tool was built to fix access/scale (VA-runnable, no
  operator login required). By porting to stateless API calls + JS structure,
  we lost the property that made it work: Claude understanding the whole
  article in context.

  V4.0 thesis: Stop fighting LLMs with regex. Use Anthropic prompt caching
  to give Claude actual context (the same context the chat version had).
  Use strict JSON throughout via Tool Use. Let JavaScript handle deterministic
  checks. Let Gemini do ONE job: semantic fact-check against research brief.
  Lock V1 bones as frozen JSON. V2 voice rewrite returns ONLY prose patches,
  never the whole object — eliminates JSON escape bugs.

  Gemini Pro consensus reached with three architectural improvements:
  1. V2 returns rewrites array only (not whole V2 object) — prevents JSON
     escape character crashes on long prose.
  2. JS injects unique IDs on every V1 object — LLMs are bad at counting,
     so patches target IDs not array indexes.
  3. Phase 3 splits: JS owns 100% of structural validation. Gemini's only
     job is semantic fact-checking against research brief. No patching
     machinery for Gemini.

  Phase 1: Research (Gemini) → JSON brief, full context cached
